# عقد ClickCounter: أول عقد ذكي لك

مرحبًا بك في أول عقد ذكي حقيقي لك! قد يبدو هذا المثال بسيطًا، لكنك على وشك تعلّم اللبنات الأساسية التي يعتمد عليها كل عقد ذكي. سنقوم ببناء **ClickCounter**، ويمكنك التفكير فيه كأنه **عداد رقمي** يعيش إلى الأبد على البلوكشين.

---

## لماذا نبدأ بهذا المثال؟

قبل أن تبني بروتوكولات **DeFi** معقدة أو أسواق **NFT**، يجب أن تفهم الأساسيات التالية:

- كيفية تخزين البيانات على البلوكشين
- كيف تقوم الدوال (Functions) بتعديل هذه البيانات
- كيف تستمر حالة العقد (State) بين المعاملات (Transactions)
- البنية الأساسية لعقد Solidity

هذا العداد البسيط يعلّمك كل ذلك. لنبدأ!

---

## العقد الكامل

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract ClickCounter {
    uint256 public counter;
    
    function click() public {
        counter++;
    }
}
```

هذا كل شيء! فقط **8 أسطر من الكود**، لكن هناك الكثير مما يحدث هنا.

---

## السطر 1: معرف الترخيص

```solidity
// SPDX-License-Identifier: MIT
```

### ما هذا؟

كل عقد ذكي يجب أن يحدد **الترخيص (License)** الخاص به. هذا يخبر الآخرين بما يُسمح لهم قانونيًا بفعله بالكود.

### لماذا MIT؟

ترخيص **MIT** من أكثر تراخيص المصادر المفتوحة تساهلًا. معناه ببساطة:

> يمكنك استخدام الكود كما تشاء، لكن لا تلمني إذا حدثت مشكلة.

### هل هو إلزامي؟

ليس إلزاميًا تقنيًا، لكن مترجم **Solidity** سيعطيك تحذيرًا إذا لم تضفه. لذلك من الأفضل إضافته.

### تراخيص شائعة أخرى

- `GPL-3.0` — مصدر مفتوح لكن أي كود مشتق يجب أن يكون مفتوح المصدر أيضًا.
- `UNLICENSED` — جميع الحقوق محفوظة.
- `Apache-2.0` — مشابه لـ MIT لكنه يوفر حماية للبراءات.

## السطر 2: توجيه Pragma

```solidity
pragma solidity ^0.8.0;
```

### ماذا يفعل؟

يخبر **المترجم (Compiler)** أي إصدار من Solidity يجب استخدامه عند ترجمة العقد.

### تفسير ^0.8.0

- `0.8.0` الحد الأدنى للإصدار
- `^` تعني هذا الإصدار أو أي تحديث فرعي متوافق معه

الإصدارات المتوافقة:

```
0.8.0
0.8.1
0.8.2
...
0.8.x
```

الإصدارات غير المتوافقة:

```
0.7.x
0.9.x
```

### لماذا هذا مهم؟

لأن الإصدارات المختلفة من Solidity تحتوي على ميزات وتحسينات أمنية مختلفة.

إصدار **0.8.0** أضاف حماية تلقائية ضد **Overflow**.

### مثال

في Solidity 0.7.x قد يحدث overflow:

```solidity
uint8 x = 255;
x = x + 1; // يصبح 0
```

في Solidity 0.8.x:

```solidity
uint8 x = 255;
x = x + 1; // المعاملة تفشل
```

## تعريف العقد

```solidity
contract ClickCounter {
```

### ما هو العقد؟

يمكن اعتباره مثل **Class** في البرمجة الكائنية.

يحتوي على:

- متغيرات الحالة (State Variables)
- الدوال (Functions)
- الأحداث (Events)
- المحددات (Modifiers)

### اسم العقد

اسم العقد هنا هو:

```
ClickCounter
```

المتعارف عليه استخدام **PascalCase**.

كل ما يوجد داخل الأقواس `{}` ينتمي لهذا العقد.

---

## متغير الحالة

```solidity
uint256 public counter;
```

هذا السطر يقوم بعدة أشياء في نفس الوقت.

### نوع البيانات `uint256`

- `uint` عدد صحيح بدون إشارة
- `256` يعني أنه يستخدم 256 بت

النطاق:

```
0 → 2^256 - 1
```

وهو رقم ضخم جدًا.

### لماذا نستخدمه؟

1. لا نحتاج أرقام سالبة
2. الحجم القياسي في Solidity
3. من غير الواقعي أن يصل للحد الأقصى

### محدد الرؤية `public`

يعني أن أي شخص يمكنه **قراءة المتغير**.

كما أن Solidity تنشئ تلقائيًا **Getter Function**.

```solidity
uint256 public counter;
```

يقوم Solidity تلقائيًا بإنشاء:

```solidity
function counter() public view returns (uint256) {
    return counter;
}
```

لذلك يمكنك من JavaScript قراءة القيمة:

```javascript
await contract.counter()
```

### أنواع الرؤية الأخرى

- `private` → داخل العقد فقط
- `internal` → داخل العقد والعقود المشتقة
- `public` → يمكن للجميع قراءته

### اسم المتغير

اسم المتغير هو:

```
counter
```

يمكن أن يكون أي اسم مثل:

```
clickCount
totalClicks
numberOfClicks
```

المتعارف عليه استخدام **camelCase**.

### القيمة الافتراضية

عند نشر العقد يبدأ `counter` بالقيمة:

```
0
```

القيم الافتراضية في Solidity:

| النوع | القيمة |
|------|------|
| uint | 0 |
| bool | false |
| address | 0x0000000000000000000000000000000000000000 |
| string | "" |

## دالة click

```solidity
function click() public {
    counter++;
}
```

### تعريف الدالة

- `function` كلمة تعريف الدالة
- `click` اسم الدالة
- `()` المعاملات
- `public` يمكن لأي شخص استدعاؤها

### جسم الدالة

```solidity
counter++;
```

وهي اختصار لـ:

```
counter = counter + 1
```

### ماذا يحدث عند استدعاء الدالة؟

1. يتم إرسال **Transaction**
2. تقوم **EVM** بتنفيذ الكود
3. يتم زيادة `counter`
4. يتم حفظ القيمة الجديدة على البلوكشين
5. يتم تأكيد المعاملة

### تكلفة Gas

كل مرة يتم استدعاء:

```
click()
```

يتم دفع **Gas** لأن الشبكة تحتاج إلى:

- تنفيذ الكود
- تخزين الحالة
- نشر التحديث عبر الشبكة

## مثال عملي

### نشر العقد

```javascript
const contract = await ClickCounter.deploy();
await contract.deployed();
```

القيمة تصبح:

```
counter = 0
```

### أول نقرة

```javascript
await contract.click();
```

```
counter = 1
```

### قراءة العداد

```javascript
const value = await contract.counter();
```

هذه عملية **قراءة** ولا تكلف Gas.

### عدة نقرات

```javascript
await contract.click(); // 2
await contract.click(); // 3
await contract.click(); // 4
```

كل معاملة يتم حفظها بشكل دائم على البلوكشين.

## أسئلة شائعة

### ماذا لو وصل العداد للحد الأقصى؟

سيحتاج:

```
2^256 نقرة
```

ولو نقرت مرة كل ثانية سيستغرق أكثر من عمر الكون.

### هل يمكن إعادة العداد للصفر؟

ليس في هذا العقد، لكن يمكن إضافة:

```solidity
function reset() public {
    counter = 0;
}
```

### من يمكنه استدعاء click؟

أي شخص لأنها `public`.

### هل قراءة العداد تكلف Gas؟

لا.

القراءة مجانية، بينما **الكتابة فقط** تكلف Gas.

## تطوير العقد

### إضافة Reset

```solidity
function reset() public {
    counter = 0;
}
```

### إضافة Decrement

```solidity
function decrement() public {
    require(counter > 0, "Counter is already at zero");
    counter--;
}
```

### تتبع المستخدمين

```solidity
mapping(address => uint256) public clicksByUser;

function click() public {
    counter++;
    clicksByUser[msg.sender]++;
}
```

### إضافة Event

```solidity
event Clicked(address indexed user, uint256 newCount);

function click() public {
    counter++;
    emit Clicked(msg.sender, counter);
}
```
