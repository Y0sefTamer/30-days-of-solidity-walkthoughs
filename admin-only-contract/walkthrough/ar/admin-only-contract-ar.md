# عقد خاصة بالمالك: أول نظام تحكم في الوصول لديك

مرحباً بعودتك! اليوم سنتناول شيئاً بالغ الأهمية في تطوير العقود الذكية: **التحكم في الوصول**.

فكر في الأمر — لا تريد أن يستطيع أي شخص إيقاف عقدك، أو سك توكنات بلا حدود، أو تفريغ الخزينة، صحيح؟ هنا يأتي دور مفهوم الدوال "الخاصة بالمالك".

## المشكلة التي نحلها

تخيّل أنك تبني صندوق كنز رقمي على البلوكشين. تريد:

- أن يكون أنت فقط (المالك) من يضيف الكنز إلى الصندوق
- أن تكون أنت فقط من يوافق على من يمكنه سحب الكنز
- ولكن أي شخص توافق عليه يجب أن يستطيع سحب المبلغ الموافق عليه

بدون تحكم وصول مناسب، يمكن لأي شخص استدعاء دوال الإدارة وإحداث الفوضى. دعنا نبني نظاماً يمنع ذلك.

---

## إعداد الملكية

أولاً، نحتاج لتحديد من هو المالك. يحدث هذا عند نشر العقد:

```solidity
address public owner;

constructor() {
    owner = msg.sender;
}
```

**ما الذي يحدث هنا؟**

- `msg.sender` هو متغير خاص يمثل من يستدعي الدالة
- في `constructor`، `msg.sender` هو من نشر العقد
- نحفظ ذلك العنوان كـ `owner`

هذا إعداد لمرة واحدة. بعد النشر، يتم تثبيت المالك. (إلا إذا أضفت دالة لنقل الملكية، وهو ما سنستعرضه لاحقاً!)

---

## سحر الـ Modifiers

هنا يصبح الوضع ممتعاً. بدلاً من كتابة نفس فحص الصلاحية في كل دالة إدارة، ننشئ **modifier** قابل لإعادة الاستخدام يعمل كحارس بوابة:

```solidity
modifier onlyOwner() {
    require(msg.sender == owner, "Access denied: Only the owner can perform this action");
    _;
}
```

**نفكك هذا:**

1. **`modifier onlyOwner()`** – نعرّف modifier جديد باسم `onlyOwner`
2. **`require(msg.sender == owner, ...)`** – هذا يفحص إذا كان المستدعي هو المالك. إذا لم يكن، تُلغى المعاملة برسالة خطأ
3. **`_;`** – هذا عنصر نائب يقول "نفِّذ شفرة الدالة هنا"

فكر في `_;` كعلامة مرجعية. يتم تشغيل الـ modifier أولاً، يفحص الأذونات، وإذا كان كل شيء صحيحاً، ينفّذ الدالة عند موقع `_;`.

**لماذا هذا قوي؟** بدلاً من تكرار عبارة `require` نفسها في 10 دوال مختلفة، تضيف `onlyOwner` إلى كل واحدة. نظيف، قابل للصيانة، وأقل عرضة للأخطاء.

---

## استخدام الـ Modifier: إضافة الكنز

لنرى الـ modifier في التطبيق:

```solidity
uint256 public treasureAmount;

function addTreasure(uint256 amount) public onlyOwner {
    treasureAmount += amount;
}
```

**ماذا يحدث عندما ينادي أحدهم هذه الدالة؟**

1. يتم تشغيل modifier `onlyOwner` أولاً
2. يفحص: "هل `msg.sender` يساوي `owner`؟"
3. إذا كانت الإجابة نعم → يواصل إلى جسم الدالة ويضيف الكنز
4. إذا كانت لا → تُلغى المعاملة مع "Access denied..."

بدون الـ modifier، يمكن لأي شخص أن ينادي `addTreasure(1000000)` ويزيد الكنز في الصندوق. معه، فقط المالك يمكنه.

---

## الموافقة على السحوبات

إليك حالة أكثر تعقيداً — يمكن للمالك الموافقة على مبالغ محددة لأشخاص معينين:

```solidity
mapping(address => uint256) public withdrawalAllowance;

function approveWithdrawal(address recipient, uint256 amount) public onlyOwner {
    require(amount <= treasureAmount, "Not enough treasure available");
    withdrawalAllowance[recipient] = amount;
}
```

**النمط:**

1. **دالة خاصة بالمالك** – فقط المالك يمكنه الموافقة على السحوبات
2. **فحص الأمان** – لا يمكن الموافقة على أكثر مما في الصندوق
3. **تسجيل المبلغ المسموح به** – نخزن كم يمكن لهذا الشخص سحبه

هذا مشابه لكيفية عمل الدالة `approve()` في توكنات ERC-20!

---

## سحب الكنز: وصول من مستويين

الآن الجزء المثير — دالة تحتوي منطقاً مختلفاً للمالك مقابل المستخدمين العاديين:

```solidity
function withdrawTreasure(uint256 amount) public {
    if (msg.sender == owner) {
        // يمكن للمالك سحب أي مبلغ
        require(amount <= treasureAmount, "Not enough treasury available for this action.");
        treasureAmount -= amount;
        return;
    }

    // يمكن للمستخدمين العاديين سحب المبلغ الموافق عليه فقط
    require(amount <= withdrawalAllowance[msg.sender], "You don't have approval for this amount");
    require(amount <= treasureAmount, "Not enough treasure in the chest");

    withdrawalAllowance[msg.sender] -= amount;
    treasureAmount -= amount;
}
```

**مساران داخل هذه الدالة:**

**المسار 1 (المالك):**

- التحقق من وجود ما يكفي من الكنز
- سحب أي مبلغ
- انتهى!

**المسار 2 (المستخدم العادي):**

- التحقق من وجود موافقة لهذا المبلغ
- التحقق من وجود ما يكفي من الكنز
- خصم من المسموح له
- خصم من الصندوق

لاحظ أننا لا نستخدم modifier `onlyOwner` هنا لأننا نريد أن يتمكن المستخدمون العاديون من الاستدعاء أيضاً — لديهم فقط صلاحيات مختلفة.

---

## لماذا هذا النمط مهم

التحكم في الوصول موجود في كل مكان في Web3:

- **عقود التوكن** – فقط المالك يمكنه سك توكنات جديدة
- **المنظمات اللامركزية (DAOs)** – فقط الحوكمة يمكنها تنفيذ الاقتراحات
- **الأسواق** – فقط المالك يمكنه إيقاف التداول
- **العقود القابلة للترقية** – فقط المسؤول يمكنه ترقية التنفيذ

الـ modifier `onlyOwner` هو خط الدفاع الأول ضد الوصول غير المصرح به.

---

## أنماط شائعة سترىها

**1. نقل الملكية:**

```solidity
function transferOwnership(address newOwner) public onlyOwner {
    require(newOwner != address(0), "New owner cannot be zero address");
    owner = newOwner;
}
```

**2. أدوار متعددة:**

```solidity
mapping(address => bool) public admins;

modifier onlyAdmin() {
    require(admins[msg.sender], "Not an admin");
    _;
}
```

**3. قيود زمنية:**

```solidity
modifier afterDeadline() {
    require(block.timestamp > deadline, "Too early");
    _;
}
```

## اعتبارات أمانية

**كن حذراً مع الملكية:**

- إذا خسرت المفتاح الخاص بالمالك، تفقد السيطرة على العقد إلى الأبد
- إذا تم اختراق مفتاح المالك، يحصل المهاجم على السيطرة الكاملة
- فكر في استخدام محافظ متعددة التوقيع (multi-sig) لحسابات المالك في الإنتاج

**اختبر modifiers الخاصة بك:**

- تأكد أنها تمنع بالفعل الوصول غير المصرح به
- اختبر حالات الحافة (ماذا لو كان المالك هو `address(0)`؟)
- تأكد أن رسائل الخطأ واضحة

---

## ما الذي تعلمته

أصبحت الآن تفهم:

- كيفية تثبيت ملكية العقد
- كيفية إنشاء modifiers قابلة لإعادة الاستخدام
- كيفية تنفيذ التحكم في الوصول بناءً على الأدوار
- لماذا التحكم في الوصول مهم للأمان

هذه معرفة أساسية. تقريباً كل عقد حقيقي يستخدم بعض أشكال التحكم في الوصول. سترى `onlyOwner` وأنماط مشابهة في كل مكان.

## تحدَّ نفسك

جرّب توسيع هذا العقد:

- أضف دالة `renounceOwnership()` التي تجعل المالك `address(0)`
- أنشئ modifier `onlyApproved` للمستخدمين الذين لديهم مخصص غير صفري
- أضف سجل سحوبات باستخدام الأحداث
- نفِّذ نظام أذونات متعدد المستويات (مالك، مسؤول، مستخدم)

في المرة القادمة، سنجمع التحكم في الوصول مع منطق أكثر تعقيداً. أنت تبني أساساً قوياً!
