# عقد المزاد (AuctionHouse Contract): بناء نظام العروض الخاص بك الأول

أهلاً وسهلاً في أحد أكثر العقود إثارة التي ستبنيها! اليوم سننشئ نظام مزاد كاملاً - فكر في `eBay`، لكن على البلوكتشين. يمكن للمستخدمين تقديم عروض والتنافس على العناصر، والعرض الأعلى يفوز.

## لماذا المزادات على البلوكتشين؟

الاجترارات التقليدية على الإنترنت بها مشاكل:

- **مشاكل الثقة** - هل يمكن للمنصة التلاعب بالعروض؟
- **الشفافية** - هل يمكنك التحقق من أن العرض الأعلى حقيقي؟
- **المركزية** - شركة واحدة تسيطر على كل شيء

مزادات البلوكتشين تحل هذه المشاكل. كل عرض عام وقابل للتحقق ودائم. لا أحد يستطيع الغش. يتم فرض القواعد بالكود، وليس بواسطة شركة قد يكون لديها تضارب مصالح.

## ما الذي سننشئه؟

نظام مزاد حيث:

- يقوم المالك بإنشاء مزاد لعنصر ما بحد زمني
- يمكن لأي شخص تقديم عروض خلال فترة المزاد
- يمكن للمستخدمين زيادة عروضهم الخاصة
- بعد انتهاء الوقت، يمكن إنهاء المزاد
- العرض الأعلى يفوز

دعونا نبنيه خطوة بخطوة!

## إعداد حالة المزاد

أولاً، نحتاج إلى تتبع جميع المعلومات حول مزادنا:

```solidity
address public owner;           // من أنشأ المزاد
string public item;             // ما الذي يتم بيعه
uint public auctionEndTime;     // متى ينتهي تقديم العروض
address private highestBidder;  // الفائز الحالي
uint private highestBid;        // مبلغ العرض الفائز
bool public ended;              // هل انتهى المزاد؟
mapping(address => uint) public bids;  // عرض كل شخص الحالي
address[] public bidders;       // قائمة جميع المشاركين
```

**لماذا هذه المتغيرات؟**

- **`owner`** - الشخص الذي نشر العقد وينظم المزاد
- **`item`** - وصف مثل "NFT نادر" أو "غيتار عتيق"
- **`auctionEndTime`** - `timestamp` (الوقت). بعد هذا الوقت، لا يُسمح بتقديم عروض جديدة
- **`highestBidder` & `highestBid`** - تتبع من يفوز
- **`ended`** - منع إنهاء المزاد عدة مرات
- **`bids` mapping** - لكل عنوان مبلغ عرضه الحالي
- **`bidders` array** - يُفيد للمرور عبر جميع المشاركين لاحقاً

لاحظ أن `highestBidder` و `highestBid` هما `private`. لا نريد للناس التطفل على من يفوز حتى ينتهي المزاد (رغم أنهم يمكنهم معرفة ذلك من `Events` أو بالاتصال بـ `getWinner()` بعد انتهاء المزاد).

## المُنشئ (Constructor): بدء المزاد

عند نشر هذا العقد، تقوم بإعداد المزاد:

```solidity
constructor(string memory _item, uint _biddingTime) {
    owner = msg.sender;
    item = _item;
    auctionEndTime = block.timestamp + _biddingTime;
}
```

**كيف يعمل:**

- **`msg.sender`** - أنت (الشخص الذي ينشر العقد) تصبح المالك
- **`_item`** - تحدد ما الذي يتم بيعه
- **`_biddingTime`** - كم عدد الثواني التي سيستمر المزاد (مثلاً 86400 لمدة 24 ساعة)

**`block.timestamp`** هو متغير خاص يعطيك الوقت الحالي على البلوكتشين بالثواني منذ `Unix epoch` (الأول من يناير 1970). بإضافة `_biddingTime` إليه، نضع موعداً نهائياً.

**مثال:** إذا قمت بالنشر في الوقت 1700000000 مع `_biddingTime = 3600` (ساعة واحدة)، ينتهي المزاد في الوقت 1700003600.

## تقديم عرض (Placing a Bid): قلب المزاد

هذا هو المكان الذي يحدث فيه الحدث:

```solidity
function bid(uint amount) external {
    require(block.timestamp < auctionEndTime, "Auction has already ended.");
    require(amount > 0, "Bid amount must be greater than zero.");
    require(amount > bids[msg.sender], "New bid must be higher than your current bid.");

    if (bids[msg.sender] == 0) {
        bidders.push(msg.sender);
    }

    bids[msg.sender] = amount;

    if (amount > highestBid) {
        highestBid = amount;
        highestBidder = msg.sender;
    }
}
```

**دعنا نمرّ عبر هذا خطوة بخطوة:**

### الخطوة الأولى: فحوصات التحقق

```solidity
require(block.timestamp < auctionEndTime, "Auction has already ended.");
```

تحقق من أننا لا نزال في فترة تقديم العروض. إذا انتهى الوقت، ارفض العرض.

```solidity
require(amount > 0, "Bid amount must be greater than zero.");
```

لا يُسمح بعروض صفرية. عليك أن تقدم عرضاً حقيقياً!

```solidity
require(amount > bids[msg.sender], "New bid must be higher than your current bid.");
```

إذا كنت قد قدمت عرضاً من قبل، يجب أن يكون عرضك الجديد أعلى. هذا يمنع الرسائل غير المرغوب فيها ويضمن أن تصعد العروض فقط.

### الخطوة الثانية: تتبع المشاركين الجدد

```solidity
if (bids[msg.sender] == 0) {
    bidders.push(msg.sender);
}
```

إذا كان هذا عرضك الأول (عرضك الحالي هو 0)، أضفك إلى قائمة المشاركين. هذه الطريقة يمكننا تتبع جميع المشاركين.

### الخطوة الثالثة: تسجيل العرض

```solidity
bids[msg.sender] = amount;
```

حدّث مبلغ عرضك. هذا يستبدل عرضك السابق.

### الخطوة الرابعة: تحقق من أنك الزعيم الجديد

```solidity
if (amount > highestBid) {
    highestBid = amount;
    highestBidder = msg.sender;
}
```

إذا كان عرضك أعلى من الحالي، تصبح الزعيم الجديد!

**ملاحظة مهمة:** هذا مزاد مبسط. في التطبيق الحقيقي، ستريد التعامل مع تحويلات `ETH` الفعلية، استرجاع الأموال للمستخدمين الذين تم تجاوز عروضهم، وأكثر. لكن هذا يعلم المنطق الأساسي بشكل جميل.

## إنهاء المزاد

بمجرد انتهاء الوقت، يحتاج شخص ما إلى إنهاء المزاد رسمياً:

```solidity
function endAuction() external {
    require(block.timestamp >= auctionEndTime, "Auction hasn't ended yet.");
    require(!ended, "Auction end already called.");
    ended = true;
}
```

**لماذا هذه الفحوصات؟**

1. **فحص الوقت** - لا يمكنك إنهاء المزاد مبكراً. المشاركون يحتاجون الفترة الكاملة
2. **فحص الإنهاء السابق** - يمنع استدعاء هذه الدالة عدة مرات

**من يمكنه استدعاء هذا؟** أي شخص! لاحظ أنه لا يوجد معدِّل `onlyOwner`. بمجرد انتهاء الوقت، يمكن لأي شخص إنهاء المزاد. هذا تصميم جيد - لا تريد أن يتعطل المزاد إذا اختفى المالك.

ضبط `ended = true` مهم لأنه يمنع أي حالات غريبة ويوضح أن المزاد انتهى.

## مشاهدة الفائز

بعد انتهاء المزاد، يمكن لأي شخص التحقق من الفائز:

```solidity
function getWinner() external view returns (address, uint) {
    require(ended, "Auction has not ended yet.");
    return (highestBidder, highestBid);
}
```

**نقاط مهمة:**

- **دالة `view`** - لا تعدّل الحالة، فقط تقرأ البيانات. لا تكلفة `Gas` عند استدعاؤها خارجياً!
- **تُرجع قيمتين** - عنوان الفائز والعرض الفائز
- **تتطلب انتهاء المزاد** - لا يمكنك التطفل على الفائز بينما تقديم العروض لا يزال جارياً

هذه الدالة تعيد أكثر من قيمة واحدة (tuple). في `JavaScript`، يمكنك فصلها هكذا:

```javascript
const [winner, winningBid] = await contract.getWinner();
```

## سيناريوهات من الحياة الحقيقية

**السيناريو الأول: ثلاثة مشاركين**

1. أليس تقدم عرضاً بـ 100 → هي صاحبة أعلى عرض
2. بوب يقدم عرضاً بـ 150 → بوب يتقدم
3. أليس تقدم عرضاً بـ 200 → أليس تستعيد الصدارة
4. ينتهي الوقت → أليس تفوز بـ 200

**السيناريو الثاني: عرض في آخر لحظة**

1. تشارلي يقدم عرضاً بـ 100 في البداية
2. لا يحدث شيء لساعات...
3. ديانا تقدم عرضاً بـ 500 مع 10 ثواني متبقية
4. ينتهي الوقت → ديانا تفوز (لا وقت للرد)

هذا يُسمى "sniping" (الرماية) وهي شائعة في المزادات. بعض المنصات تضيف آليات "anti-snipe" تمدد الوقت إذا وافقت عروض بالقرب من النهاية.

## ما الذي ينقص؟ (ولماذا)

هذا عقد تعليمي، لذا بسطنا بعض الأشياء:

**1. لا توجد معالجة `ETH` فعلية** - في الإنتاج، ستستخدم `payable` و `msg.value` للتعامل مع الأموال الحقيقية

**2. لا يوجد استرجاع أموال** - عندما يتم تجاوز عرضك، يجب أن تحصل على `ETH` الخاص بك تلقائياً

**3. لا يوجد حد أدنى لزيادة العرض** - المزادات الحقيقية غالباً تتطلب زيادة العروض بنسبة لا تقل عن X٪

**4. لا توجد نسبة احتياطية** - العديد من المزادات لها حد أدنى للسعر يجب أن تصل إليه العناصر

**5. لا نمط سحب الأموال** - يجب أن يكون بإمكان الفائزين المطالبة بالعنصر، والبائعون يجب أن يحصلوا على أموالهم

سنغطي هذه الأنماط في العقود المستقبلية!

## المفاهيم الرئيسية التي تعلمتها

**1. منطق يعتمد على الوقت** - استخدام `block.timestamp` للمواعيد النهائية

**2. إدارة الحالة** - تتبع حالة المزاد المعقدة عبر متغيرات متعددة

**3. أنماط التحقق** - عبارات `require()` متعددة لفرض القواعد

**4. عام مقابل خاص** - إخفاء `highestBidder` حتى الوقت المناسب

**5. دوال `View`** - قراءة البيانات دون تكاليف `Gas`

---

## اعتبارات الأمان

**معالجة `block.timestamp`:** يمكن للمُعدِّنين التلاعب بـ `block.timestamp` بضع ثوان. بالنسبة للمزادات ذات القيمة العالية، قد يكون هذا مهماً. فكر في استخدام أرقام `block` بدلاً من ذلك، أو اقبل المخاطرة الصغيرة.

**`Front-running` (التقدم مسبقاً):** على البلوكتشين العام، يمكن للآخرين رؤية عرضك قبل تعدينه وتقديم عرض أعلى بمزيد من `Gas` ليتم تعدينه أولاً. هذه مشكلة بلوكتشين أساسية.

**الحرمان من الخدمة:** إذا قمت باسترجاع الأموال للمستخدمين المتجاوزة عروضهم تلقائياً، فقد يرفض عقد خبيث الاسترجاع ويكسر مزادك. استخدم نمط السحب بدلاً من ذلك (المستخدمون يسحبون الأموال بدلاً من أن تدفعها أنت).

## تحدِّ نفسك

وسعّ هذا العقد:

1. **أضف معالجة `ETH`** - اجعل `bid()` قابلة للدفع وتتبع الأموال الحقيقية
2. **طبّق استرجاع الأموال** - أنشئ دالة `withdraw()` للمستخدمين المتجاوزة عروضهم
3. **أضف نسبة احتياطية** - المزاد ينجح فقط إذا كان أعلى عرض يلبي الحد الأدنى
4. **أنشئ مزادات متعددة** - استخدم `structs` و `mappings` لتشغيل مزادات كثيرة بالتزامن
5. **أضف أحداث (Events)** - أصدر `BidPlaced`، `AuctionEnded`، إلخ للتكامل مع الواجهة الأمامية

لقد بنيت للتو الأساس الذي تعتمد عليه أسواق البلوكتشين اللامركزية. هذا النمط نفسه يُشغّل مزادات `NFT` وبيع أسماء النطاقات وأكثر. مبروك!
